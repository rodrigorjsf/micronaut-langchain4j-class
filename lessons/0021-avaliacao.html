Avaliação: medir o assistente sem framework (LLM-as-judge no Bedrock)

"Parece bom" não passa numa auditoria bancária. E o Java não tem RAGAS. Esta lição mostra como medir o assistente com honestidade: os dois eixos da avaliação, o juiz LLM construído no Bedrock e o gate de regressão em CI.

Por que isto importa para a sua missão — num banco regulado, cada resposta que move dinheiro ou toca dado de cliente precisa ser auditável. Você não pode confiar em "vibes": precisa de um conjunto de testes versionado e de um gate que falha o build quando a qualidade regride. E como o ecossistema de avaliação é Python, você vai construir o avaliador — com as peças que já tem no Bedrock + LangChain4j.

1 · Por que "vibes" não basta

Avaliar não é abrir o chat e achar a resposta boa. É um conjunto de testes offline versionado e commitado — como fixtures de teste: triplas pergunta → resposta-ouro → documentos-ouro esperados, uma por escopo de conta (nacional/global, por produto). Sem isso, um ajuste de prompt ou uma troca de modelo regride em silêncio uma resposta de movimentação ou de KYC — e você só descobre em produção, com o cliente.

2 · Os dois eixos (medidos separadamente)

RAG (Lição 5) tem duas qualidades independentes. Misturá-las esconde a causa de uma falha:

EixoPerguntaMétricasComo se mede
RecuperaçãoVieram os chunks certos?recall@k, precision@k, MRRDeterminístico — matemática de conjuntos sobre os doc-ids ouro. Sem LLM.
GeraçãoA resposta é fiel e relevante?faithfulness, answer relevancyLLM-as-judge — um modelo forte pontua contra uma rubrica.

Fidelidade (faithfulness/groundedness) = a resposta está aterrada nos chunks, sem inventar. Relevância = responde à pergunta. As definições são agnósticas de linguagem (a referência é o RAGAS) — você as implementa em Java, não as importa.

3 · A realidade Java (honesta)

★ Insight — Java não tem camada de avaliação nativa. O ecossistema Python entrega graders de fábrica (RAGAS, DeepEval, TruLens); na JVM nenhum roda in-process. Mas o gap é de conveniência, não de capacidade: LLM-as-judge é só um prompt + um modelo forte + uma rubrica — que você já tem via Bedrock. É a mesma honestidade da Lição 19.

4 · O padrão honesto: LLM-as-judge no Bedrock

Use o AiServices (Lição 6) com o juiz retornando um record tipado — nunca Map (a saída por Map é degenerada, Lição 10):

record Verdict(int faithfulness, int relevancy, String rationale) {}   // 0..5 + justificativa

interface RagJudge {
    @SystemMessage("You are a strict evaluator of a banking assistant. "
        + "Score faithfulness (answer fully supported by the context, no invented facts) "
        + "and relevancy (answer addresses the question) from 0 to 5. Be conservative.")
    @UserMessage("Question: {{q}}\nContext: {{ctx}}\nAnswer: {{a}}")
    Verdict judge(@V("q") String question, @V("ctx") String context, @V("a") String answer);
}

RagJudge judge = AiServices.create(RagJudge.class, judgeModel);   // judgeModel = BedrockChatModel
Auditabilidade (o que torna o juiz defensável): fixe o juiz num inference profile Claude conhecido, ponha a temperatura em 0, versione a rubrica e guarde o rationale de cada nota. Um juiz LLM é não-determinístico — então, para gates de conformidade duros, pareie-o com as métricas determinísticas de recuperação (recall@k não mente).

5 · O gate de CI

Rode o conjunto de eval offline a cada mudança (de prompt, modelo, chunking): falhe o build se faithfulness ou recall@k caírem abaixo do limiar. O promptfoo pode hospedar o conjunto contra o endpoint Micronaut; o juiz fica dentro do stack (Bedrock). Assim, um bump de modelo não regride às escondidas uma resposta que move dinheiro — a regressão vira um build vermelho, não um incidente.

6 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Por que "vibes" não basta para avaliar num banco regulado?

Eval = conjunto offline versionado + gate de CI que falha o build na regressão.

2. Para que serve o ScoringModel do langchain4j?

É reranker (eixo recuperação), não avaliador de resposta. O juiz você constrói.

3. Como tornar o juiz LLM auditável num banco?

Juiz fixo + temp 0 + rubrica versionada + rationale guardado; pareado com recall@k.
