Agent Skills no Java: instruções sob demanda com langchain4j-skills

O assistente do banco precisa saber muita coisa por domínio (Pix, cartão, investimentos, financiamento…). Empilhar tudo no system prompt não escala. As skills empacotam esse conhecimento e deixam o modelo puxá-lo sob demanda — e tudo isso é construído sobre o mesmo ToolProvider da Lição 7.

Por que isto importa para a sua missão — cada produto do banco tem regras próprias (e nacional vs. global multiplica tudo). Carregar todas as instruções em todo turno custa tokens, polui o contexto e degrada decisões. Skills mantêm o system prompt enxuto: o modelo vê um catálogo curto e só carrega o manual do domínio quando realmente precisa — disciplina direta de orçamento de contexto (Lição 8) aplicada a instruções, não só a tools.

1 · O problema: o system prompt não escala

A Lição 7 mostrou o tool retrieval para não despejar dezenas de tools na mesa. O mesmo problema vale para instruções: o manual completo de "como fazer Pix na conta global", "como simular financiamento", "como ler o Super Extrato" — se tudo isso vive no system prompt, ele incha, encarece cada chamada (o custo N² da Lição 8) e afoga o sinal em ruído. A resposta é progressive disclosure (revelação progressiva): mostrar pouco, carregar o resto sob demanda.

2 · O que é uma skill: o formato SKILL.md

Uma skill é um diretório com um arquivo SKILL.md: um front matter YAML que declara name e description (sempre visíveis ao modelo), um corpo Markdown de instruções (o conteúdo, carregado na ativação) e, opcionalmente, arquivos de recursos (em references/, scripts/…). Segundo a especificação que o langchain4j referencia (agentskills.io), o formato foi originalmente criado pela Anthropic e liberado como padrão aberto.

# skills/pix-global/SKILL.md
---
name: pix-global
description: Limites e envio de Pix na conta GLOBAL (regras e backend distintos da nacional).
---

# Pix na conta global
Use quando o cliente perguntar sobre Pix na conta global.
1. Confirme o tipo de conta antes de qualquer ação de envio.
2. Cheque o limite com a tool de leitura; só então proponha o envio.
(referências em references/, exemplos em scripts/ — carregados sob demanda)

3 · O módulo real: dev.langchain4j:langchain4j-skills

Aterrada por verificação contra a fonte na tag 1.16.2 (o módulo é @Experimental / beta). A API pública essencial:

SímboloPapel
Skills (classe)Skills.from(...), .toolProvider() (o ToolProvider das skills), .formatAvailableSkills() (o catálogo XML para o system message)
Skill (interface)name(), description(), content(), resources(), toolProviders(); Skill.builder(). Não há anotação @Skill
FileSystemSkillLoaderloadSkills(Path) / loadSkill(Path) — lê diretórios com SKILL.md
ClassPathSkillLoadero mesmo, mas do classpath (inclusive dentro de JARs)
SkillResource, ActivateSkillToolConfig, ReadResourceToolConfigrecurso sob demanda; e a configuração (nomes/descrições) dos tools activate_skill / read_skill_resource
★ Pegadinha verificada — não existe anotação @Skill e não existe método AiServices.skills(...). Skill é uma interface; o registro é via .toolProvider(skills.toolProvider()). E cuidado para não confundir com o módulo irmão langchain4j-experimental-skills-shell (classe ShellSkills), que executa shell: o langchain4j-skills carrega tudo em memória, sem acesso ao filesystem em tempo de inferência e sem execução de código arbitrário.

4 · Como liga ao AiService (sobre o ToolProvider)

Dois passos (verbatim do javadoc de Skills): injetar o catálogo no system message e registrar o ToolProvider das skills.

// 1) Carrega as skills (cada uma = um diretório com um SKILL.md)
Skills skills = Skills.from(FileSystemSkillLoader.loadSkills(Path.of("skills")));

// 2) Liga ao AiService: catálogo no system message + tools de ativação
AssistenteBanco assistente = AiServices.builder(AssistenteBanco.class)
    .chatModel(model)                                   // BedrockChatModel (modelo-agnóstico)
    .systemMessage("Você é o assistente do banco.\n"
        + "Skills disponíveis:\n" + skills.formatAvailableSkills())
    .toolProvider(skills.toolProvider())               // expõe activate_skill / read_skill_resource
    .build();

O ToolProvider das skills é controlado por ativação: ele sempre expõe activate_skill (e read_skill_resource quando há recursos), e só revela as tools escopadas de uma skill depois que o modelo chamou activate_skill — rastreado em memória pelo atributo activated_skill nas mensagens, nunca enviado ao provedor. Ele é dinâmico (isDynamic() retorna true) quando alguma skill traz tools próprias (Skill.toolProviders()); para skills só de instruções, expõe apenas activate_skill/read_skill_resource.

★ Insight — três estratégias, um substrato. Esta é a terceira forma de controlar "o que o modelo vê", todas sobre o mesmo ToolProvider da Lição 7: (1) .tools(...) estático; (2) busca vetorial (ToolSearchStrategy); (3) ativação (skills). As duas últimas são ortogonais e compõem — os testes do módulo ligam .toolProvider(skills.toolProvider()) junto com .toolSearchStrategy(...), com activate_skill sempre visível e as tools comuns ainda searchable.

5 · A versão certa: lockstep + BOM

O artefato é @Experimental e o sufixo -betaN acompanha o core em lockstep: langchain4j-skills:X.Y.Z-betaN depende de langchain4j:X.Y.Z. Logo, a coordenada certa depende de qual core você usa — não decore um número, derive-o:

Seu coreCoordenada de skillsDe onde
1.15.1 (embarcado pelo micronaut-langchain4j 2.0.1, Lição 12)langchain4j-skills:1.15.1-beta25depende de core 1.15.1
1.16.2 (a pilha agêntica, Lição 14)langchain4j-skills:1.16.2-beta26depende de core 1.16.2
 
<dependency>
  <groupId>dev.langchain4j</groupId>
  <artifactId>langchain4j-skills</artifactId>
  <version>1.16.2-beta26</version>   <!-- ou 1.15.1-beta25 p/ o core embarcado -->
</dependency>

Melhor ainda: deixe o langchain4j-bom do seu core resolver a versão (ele lista langchain4j-skills), e não fixe o número à mão. Verifique sempre na tabela de versões: reference/versoes-2026-06. Atenção: a fonte está nas tags estáveis 1.15.1/1.16.2 — não existe tag git 1.15.1-beta25 (o beta é só do artefato).

⚠️ Honestidade de plataforma (Bedrock). O módulo é modelo-agnóstico: liga via ChatModel/AiServices/ToolProvider, e BedrockChatModel implementa ChatModel — então funciona no caminho Bedrock do banco. O único requisito é um modelo com tool calling, que o Claude no Bedrock tem. O fluxo básico (activate_skill + read_skill_resource) é o confirmado; a integração opcional com tool search sobre o Bedrock não foi verificada ponta a ponta — não a trate como garantida.

6 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Na progressive disclosure das skills, o que o modelo vê antes de ativar uma?

Só name+description (via formatAvailableSkills()) ficam visíveis; o corpo entra quando o modelo chama activate_skill.

2. Como uma skill se registra num AiService do langchain4j-skills?

Não há @Skill nem .skills(...). Skill é uma interface; o registro é pelo ToolProvider que Skills expõe.

3. Para o core 1.15.1 embarcado pelo micronaut-langchain4j, qual skills casa?

skills:X.Y.Z-betaN depende de core:X.Y.Z; o core embarcado é 1.15.1, então a coordenada é 1.15.1-beta25 (idealmente via BOM).
