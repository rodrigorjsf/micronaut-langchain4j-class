Segurança: prompt injection e o OWASP LLM Top 10 (2025)

A lição-capstone de segurança: ela costura as defesas das lições anteriores (1, 7, 9, 10, 11) no padrão de mercado — o OWASP Top 10 para Aplicações LLM, edição 2025 (v2.0) — e fixa a linha-mestra: o controle vive no código, nunca no prompt.

Por que isto importa para a sua missão — num banco, segurança não é um módulo "extra": é o que decide se o sistema pode ir a produção. Esta lição te dá um checklist auditável (o OWASP 2025) e mostra que cada item já tem uma defesa que você construiu — desde que entenda que autorização é determinística, não uma instrução ao modelo.

1 · A edição certa (corrige o mito)

A edição atual é 2025 (v2.0). Não existe edição 2026 de Aplicações LLM — o "2026" é o OWASP Top 10 for Agentic Applications (ASI Top 10, IDs ASI##), projeto separado. Novos em 2025: LLM07 System Prompt Leakage e LLM08 Vector and Embedding Weaknesses. Prompt Injection é o #1 (LLM01) nas duas edições.

2 · LLM01 Prompt Injection: direta vs. indireta

  • Direta (verbatim): "a user's prompt input directly alters the behavior of the model" — o cliente digitando um jailbreak no chat.
  • Indireta (verbatim): "an LLM accepts input from external sources, such as websites or files" — a variante RAG, a mais fácil de subestimar: instrução escondida no dado ingerido (um PDF de extrato, um campo de memo de transação, o corpo de um ticket). E "injections do not need to be human-visible/readable, as long as the content is parsed by the model".

4 · A linha-mestra: o controle vive no código, nunca no prompt

★ Insight — O OWASP diz, com todas as letras, o que esta trilha repete desde a Lição 1. LLM06 Complete mediation (verbatim): "Implement authorization in downstream systems rather than relying on an LLM to decide if an action is allowed or not." LLM06 Require user approval: human-in-the-loop para ações de alto impacto. LLM07: o system prompt não guarda segredo. Conclusão: autenticação, autorização, confirmação e idempotência são determinísticas e suas; guardrails e saída tipada reduzem risco residual, não substituem o controle. Um system prompt dizendo "só aja na conta do próprio cliente" não é um controle — o modelo pode ser persuadido a desobedecer.

5 · As duas prioridades do banco (código)

LLM06 (Excessive Agency) e LLM08/LLM02 (vazamento cross-tenant) são as mais críticas. Os controles são Java puro determinístico — não há anotação que os substitua:

// LLM05/LLM01: saída tipada limita o que o modelo pode emitir (não texto livre de transferência)
record TransferRequest(String fromAccountId, String toAccountId, long amountCents, String idempotencyKey) {}

class TransferService {                          // LLM06 "complete mediation": authz no backend, não no modelo
    Receipt execute(TransferRequest req, AuthenticatedCustomer caller, boolean humanConfirmed) {
        if (!caller.ownsAccount(req.fromAccountId()))      // LLM06 + LLM02: saída do modelo é não-confiável
            throw new ForbiddenException("caller does not own source account");
        if (!humanConfirmed)                               // LLM06 "require user approval" (HITL)
            throw new ConfirmationRequiredException();
        rateLimiter.check(caller.id());                    // LLM10: anti denial-of-wallet
        return ledger.transferIdempotent(req);             // idempotencyKey deduplica retries
    }
}

6 · Sanitização: a defesa em profundidade (LLM01 + LLM05 + ASVS)

Os controles de §5 são a fronteira. A sanitização é a camada que reduz o impacto do que atravessa — não a substitui. O OWASP é honesto sobre isso: "não está claro se há métodos infalíveis de prevenção para prompt injection; as medidas a seguir podem mitigar o impacto" (LLM01). No banco, o conteúdo não-confiável chega pela ingestão de RAG — PDFs de extrato, campos de memo de transação, corpo de ticket — o vetor indireto (LLM01 Cenário #4: "um atacante modifica um documento num repositório usado por uma aplicação RAG; quando a consulta do usuário retorna o conteúdo modificado, as instruções maliciosas alteram a saída do modelo"). Um pipeline em camadas:

CamadaO que fazFonte
1. Normalizar (na ingestão)Normalizer.normalize(s, NFKC); remova caracteres de controle e zero-width. Frustra evasão por Base64/emoji/multilíngue (LLM01 Cenário #9).Hardening padrão (não é item numerado do ASVS)
2. Segregar o conteúdo externoMarque-o explicitamente como dado, não instrução; ponha-o em tool results delimitados e codifique em JSON (delimitador inequívoco). Diga ao modelo de onde veio.OWASP LLM01 #6 ("Separate and clearly denote untrusted content"); guia da Anthropic (JSON-encode + tool results)
3. Filtrar (deny-list)Defina categorias sensíveis e varra por conteúdo não permitido — filtros semânticos + checagem de string.OWASP LLM01 #3 ("scan for non-allowed content")
4. Validação positiva (allow-list)Valide entradas contra allow-list de valores esperados (escopos de conta, enums); dado estruturado fortemente tipado, com limite de tamanho e schema.OWASP ASVS V4.0.3 — 5.1.3 / 5.1.4 / 5.2.2
5. Codificar a saída (antes de render/exec)Codificação ciente de contexto da saída do modelo antes de renderizar (HTML/JS/URL) ou usar (SQL parametrizado). A saída do modelo é entrada não-confiável do próximo sistema.OWASP LLM05 (Improper Output Handling) + ASVS 5.3.1

★ Insight — Repare na simetria: você sanitiza a entrada que vai ao modelo e codifica a saída que sai dele. As duas direções tratam o modelo como não-confiável — exatamente o zero-trust do LLM05 ("treat the model as any other user, adopting a zero-trust approach, and apply proper input validation on responses coming from the model"). Mas nenhuma camada é a tranca: a tranca é o ownsAccount / HITL / idempotência de §5. Sanitização é o cinto; a autorização é o freio.

Honestidade de fundação (caminho Bedrock): as defesas turnkey da Anthropic — um classificador Haiku para injeção, constrained decoding via output_config, mensagens system mid-conversa — vivem na API direta/plataforma da Anthropic, não no caminho Bedrock + langchain4j do banco. Aqui você constrói cada camada à mão (ou via os guardrails de entrada/saída da Lição 11). E a edição vigente é 2025 (v2.0); o ASVS citado é o V4.0.3.

7 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Qual a edição atual do OWASP Top 10 para Aplicações LLM?

A edição atual é 2025 (v2.0). O "2026" é o projeto separado de Aplicações Agênticas (ASI Top 10), não a lista de Aplicações LLM.

2. O que é injeção de prompt INDIRETA num banco?

Indireta = o modelo aceita input de fontes externas (arquivos/dados ingeridos). É a variante RAG; a instrução nem precisa ser visível a humanos, só ser parseada pelo modelo.

3. Onde vive o controle que impede o modelo de autorizar uma transferência?

LLM06 "complete mediation": autorize nos sistemas downstream, não confie no LLM. O prompt não é controle; guardrail é defesa probabilística, não autorização.
