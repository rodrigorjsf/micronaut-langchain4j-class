Java 25 para chamadas de tool em paralelo

O banco consulta a conta nacional e a global em backends distintos. Buscá-las em paralelo corta latência — e o Java 25 + LangChain4j dão o caminho GA-seguro, sem cair nos mitos de "tudo novo no 25".

Por que isto importa para a sua missão — latência percebida é experiência do cliente. Mas concorrência mal-entendida vira bug em sistema que move dinheiro. Esta lição separa o que é GA-seguro hoje (virtual threads, scoped values, o opt-in de tools concorrentes do LangChain4j) do que ainda é preview — e fixa que concorrência ≠ segurança transacional.

1 · O status REAL dos recursos do Java 25 (corrige o mito)

RecursoStatus no Java 25Flag?
Virtual ThreadsFinal desde o JDK 21 (JEP 444); permanente no 25 LTSnão
Scoped ValuesFinal no JDK 25 (JEP 506) — orElse não aceita mais nullnão
Structured ConcurrencyAinda PREVIEW (JEP 505, quinto preview)sim
Não diga "Virtual Threads finalizadas no 25" — final no 21. E a StructuredTaskScope no JDK 25 mudou: agora é aberta por fábricas estáticas StructuredTaskScope.open() / open(Joiner) — código de previews 21–24 (new StructuredTaskScope<>(), ShutdownOnFailure) não compila no 25.

2 · O default do LangChain4j é SEQUENCIAL (o fato-chave)

O AiServices executa múltiplas tools de um turno sequencialmente por padrão (verificado em ToolService.execute → executeSequentially quando executor == null). Para paralelizar, é opt-in:

AiServices.builder(...).executeToolsConcurrently(executor)   // @since 1.4.0
  • Concorre apenas quando o turno tem >1 tool call e há executor (if (executor != null && toolRequests.size() > 1)).
  • Para um ChatModel síncrono (como o BedrockChatModel), uma única tool roda na thread chamadora (executor é ignorado, de propósito). StreamingChatModel difere.

3 · Virtual threads por baixo (o fit para I/O)

O executor default do executeToolsConcurrently() é um singleton lazy de virtual-thread-per-task (DefaultExecutorProvider, padrão Holder; VirtualThreadUtils → Executors.newVirtualThreadPerTaskExecutor() em Java 21+). Fan-out de backends é I/O-bound → virtual threads são o encaixe (uma por tarefa; não faça pool).

★ Insight — Em produção, deixe o Micronaut dono do executor. Injete @Named(TaskExecutors.VIRTUAL) ExecutorService (constante "virtual") e passe-o a executeToolsConcurrently(executor); rode a chamada bloqueante fora do event loop com @ExecuteOn(TaskExecutors.VIRTUAL). Assim o lifecycle e a observabilidade do executor ficam com o framework — não com o singleton interno do LangChain4j.

4 · Scoped Values: contexto do request (final no 25) — e onde ele NÃO chega

ScopedValue (JEP 506, GA no 25) carrega identidade/escopo do request de forma imutável (mais seguro que ThreadLocal em virtual threads). É herdado pelas subtasks forkadas por uma StructuredTaskScope — mas não se propaga sozinho para as threads de um ExecutorService comum.

public static final ScopedValue<String> CUSTOMER_ID = ScopedValue.newInstance();
// ...
ScopedValue.where(CUSTOMER_ID, customerId).call(() -> assistant.chat(prompt));
★ Pegadinha verificada (vs JEP 506) — o executor default do executeToolsConcurrently() é um newVirtualThreadPerTaskExecutor() comum (seção 3), e ScopedValue só é herdado por subtasks de uma StructuredTaskScope — não pelas threads desse executor. Logo, dentro de uma tool executada concorrentemente, CUSTOMER_ID.get() lançaria NoSuchElementException. A identidade chega à tool concorrente pelo @ToolMemoryId (injetado pelo framework — Lição 7), não pelo ScopedValue. Regra prática: ScopedValue para o escopo da thread chamadora; @ToolMemoryId para identidade dentro das tools.

5 · Structured Concurrency: "para onde vai" (preview)

A StructuredTaskScope (JEP 505, preview no 25) trataria as buscas nacional+global como uma unidade (uma falha cancela as irmãs; join estruturado) — mas é preview (--enable-preview). Não ensine como GA. O primitivo GA-seguro para o fan-out do banco é o executeToolsConcurrently() (que já usa virtual threads por baixo).

// PREVIEW (javac --release 25 --enable-preview ; java --enable-preview) — JDK 25 usa open()/open(Joiner):
// try (var scope = StructuredTaskScope.open(/* Joiner */)) {
//     var nac = scope.fork(() -> nacional.saldo());
//     var glb = scope.fork(() -> global.saldo());
//     scope.join();
//     return combina(nac.get(), glb.get());
// }

6 · A fronteira: concorrência ≠ segurança transacional

★ Insight — Paralelizar é otimização de latência, NÃO de transação. Habilitar executeToolsConcurrently() é seguro para fan-out de LEITURA (saldos/extratos nacional + global em paralelo). Operações que movem dinheiro continuam exigindo idempotência + confirmação (Lições 12/14), rode qual executor rodar. Concorrência não dá atomicidade.

7 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Por padrão, o LangChain4j 1.16.2 executa múltiplas tools de um turno como?

ToolService.execute usa executeSequentially quando executor==null. O paralelo exige executeToolsConcurrently(executor) E mais de uma tool call no turno.

2. No Java 25, qual recurso ainda é preview (precisa de --enable-preview)?

VT são finais desde o 21; Scoped Values finais no 25 (JEP 506). Só a Structured Concurrency (JEP 505) segue preview, atrás de --enable-preview.

3. Dá para paralelizar uma transferência de dinheiro com executeToolsConcurrently?

Concorrência é otimização de I/O para LEITURA. Mover dinheiro exige idempotência + confirmação na sua lógica; o executor não confere atomicidade.
