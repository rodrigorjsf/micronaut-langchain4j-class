Arquitetura multi-backend do banco: agêntica, HITL e idempotência

O banco tem vários backends (conta nacional, global, cartão, investimentos) e operações que movem dinheiro. Esta lição mostra como compor isso com o módulo agêntico — e, sobretudo, onde o framework não te ajuda (e por que isso é o certo).

Por que isto importa para a sua missão — orquestrar múltiplos backends e fluxos de confirmação é o coração do assistente bancário. O módulo agêntico do LangChain4j dá os blocos de composição — mas a segurança de mover dinheiro (idempotência, confirmação) continua sendo sua, na camada determinística. Confundir isso é caro.

1 · O módulo agêntico (e a armadilha de versão)

Verificado no Maven Central: o módulo é publicado só como dev.langchain4j:langchain4j-agentic:1.16.2-beta26 — não existe um 1.16.2 puro (a tag git serve o código, mas não há artefato). Já o langchain4j-bedrock é release puro 1.16.2. No mesmo build, versões diferentes: bedrock=1.16.2, agentic=1.16.2-beta26. O -beta sinaliza API instável — fixe a versão exata.

AgenticServices (public class, não final) é o ponto de entrada, com fábricas estáticas: sequenceBuilder(), parallelBuilder(), loopBuilder(), conditionalBuilder(), supervisorBuilder(), plannerBuilder(), humanInTheLoopBuilder(), e createAgenticSystem(Class[, ChatModel][, AgentConfigurator]).

2 · O seam para o Bedrock + @Agent vs @V/@P

As sobrecargas que recebem ChatModel são o encaixe do BedrockChatModel (que implements ChatModel) — Claude via Bedrock, sem SDK Anthropic direto.

★ Pegadinha de ensino — a interface do agente usa @V (lê variáveis do AgenticScope — saídas de agentes anteriores): @Agent("...") String withdraw(@V("user") String u, @V("amount") Double a). A classe de tool usa @P (descreve parâmetros da @Tool para o modelo): @Tool("...") Double withdraw(@P("user name") String u, @P("amount") Double a). Trocar um pelo outro não funciona como esperado.

3 · Human-in-the-loop: genérico, sem semântica de dinheiro

Duas formas: a anotação @HumanInTheLoop (declarativa) e o record HumanInTheLoop (programática). Ambas são um mecanismo genérico de "perguntar ao usuário" — não há semântica de 'confirmação de dinheiro' embutida.

O portão approve/deny é o seu responseProvider, que em produção precisa carregar o contexto da operação (valor, contas origem/destino, idempotency key) para uma decisão informada. O exemplo de 1 argumento do tutorial é didaticamente ingênuo — não o copie para produção.

4 · Idempotência + confirmação = ARQUITETURA, não API (a joia)

★ Insight — O framework não tem primitivo de idempotência nem de confirmação de dinheiro (grep exaustivo nos 141 arquivos do langchain4j-agentic 1.16.2: zero ocorrências de "idempoten"; só literais de exemplo "user-approval"/"approved" em javadoc). E isso é correto: mover dinheiro com segurança é responsabilidade determinística da sua camada de tool/backend — dedupe por transferId/idempotency key, e o gate de confirmação na sua lógica (a LoopAgentService.exitCondition serve de porta determinística de "retry-até-aprovar"). O tema de sempre: o modelo decide e sugere; o seu código valida, autoriza e efetiva (Lições 1, 9, 11).

5 · Nacional vs. global: suas tools, seu roteamento

Nada no módulo agêntico endereça o split nacional/global — modele como tools distintas (uma fala com cada backend) ou roteie com conditionalBuilder/supervisorBuilder. O paralelismo de leitura entra via parallelBuilder (ou o executeToolsConcurrently da Lição 13).

6 · Esboço (com a assimetria de versão)

// bedrock = 1.16.2 (release)  |  agentic = 1.16.2-beta26 (experimental) — versões DIFERENTES no build
ChatModel claudeViaBedrock = BedrockChatModel.builder()
        .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0").region(Region.US_EAST_1).build();

class BankTool {                                  // @Tool usa @P; idempotência é SUA
    @Tool("Withdraw amount for a user; returns new balance")
    Double withdraw(@P("user name") String user, @P("amount") Double amount) {
        // escolhe backend nacional vs global; aplica dedupe por idempotency key
        return /* novo saldo */ 0.0;
    }
}

interface BankAgent {                             // interface de agente usa @V
    @Agent("A banker that withdraws from an account")
    String withdraw(@V("user") String user, @V("amount") Double amount);

    @HumanInTheLoop(description = "Confirm money-moving operation", outputKey = "approval")
    String confirm(@V("user") String user, @V("amount") Double amount);
}

BankAgent bank = AgenticServices.createAgenticSystem(BankAgent.class, claudeViaBedrock);
★ Esboço de anotações — NÃO auto-compõe — createAgenticSystem sobre dois métodos @Agent soltos (sem anotação de orquestração) não impõe ordem: aqui confirm não "trava" withdraw. Para ordenar, declare a composição explícita — AgenticServices.sequenceBuilder() ou a anotação @SequenceAgent (ambas verificadas em 1.16.2). E o portão que de fato protege o dinheiro (aprovar/negar) é a sua camada determinística: o responseProvider do HITL (seção 3) carregando o contexto da operação mais o dedupe por idempotency key na tool (seção 4) — porque o HITL é um "perguntar" genérico, sem semântica de dinheiro embutida.

7 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Como adicionar o módulo agêntico do LangChain4j ao build?

No Maven Central, o agentic só existe como 1.16.2-beta26; o bedrock é release puro 1.16.2. No mesmo build você usa as duas versões diferentes.

2. Quem garante idempotência e confirmação numa transferência?

Não há primitivo de idempotência/confirmação no framework (1.16.2). É arquitetura determinística sua; o HITL só pergunta, sem semântica de dinheiro.

3. Na interface de um @Agent, como referenciar variáveis do escopo?

A interface do agente usa @V (variáveis do AgenticScope, saídas de agentes anteriores). @P é para parâmetros de @Tool. São anotações distintas.
