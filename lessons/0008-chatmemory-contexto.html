ChatMemory e a gestão da janela de contexto

O tema crítico: como reter histórico sem deixar o custo N² (Lição 2) explodir num orçamento de 200K tokens — e como o prompt caching do Claude na Bedrock vira a alavanca direta contra esse custo.

Por que isto importa para a sua missão — seu assistente bancário roda em Claude Sonnet (200K) via AWS Bedrock e reenvia o histórico a cada turno. Sem gestão de contexto e caching, a conta de inferência cresce em N² e o produto deixa de ser viável. Esta lição é, antes de tudo, engenharia de custo.

1 · O problema, consolidado

Três fatos já provados se combinam: o modelo é sem estado (Lição 1) → reenviamos o histórico a cada turno → o custo cresce ~N² (Lição 2) → e a janela é finita: 200K (entrada + saída, saída até 64K). A "mesa" enche, e cada token reenviado custa.

2 · O orçamento de 200K: quem disputa a janela

ComponenteTamanho (ilustrativo)Natureza
System prompt + descrições de dezenas de tools~6.000Fixo (todo turno)
RAG: regras de produto~3.000Por pergunta
Histórico da conversa~10.000 e crescendoAcumula (N²)
Resultado de tools (extrato, fatura)~1.500 cada — podem ser enormesPor chamada
Reservado p/ saídaaté 64KDisputa o mesmo orçamento

★ Insight — Caber em 200K quase nunca é o problema — o custo é. Um turno típico usa ~20K de 200K: sobra espaço. A dor é que o bloco fixo de ~6K (system + tools) é reenviado a cada turno e o histórico cresce em N². Em pt-BR (que ainda infla tokens) × milhões de turnos/dia, é a maior conta recorrente. Gestão de contexto, num banco, é primariamente engenharia de custo.

4 · Prompt caching: a alavanca contra o N²

Fato verificado (fonte 1.16.2 + docs AWS): no Bedrock, a leitura de cache é muito mais barata. A doc do langchain4j-bedrock/AWS cita números ilustrativos — até ~90% de economia no custo de leitura e ~85% menos latência —, mas o ganho real depende da taxa de reúso e da carga: meça no seu caso, não trate como garantia. A escrita pode custar mais que entrada normal — só compensa com reúso frequente dentro do TTL. O mínimo cacheável varia por modelo: 4.096 tokens no Sonnet 4.5, 1.024 no Sonnet 4.6. Máximo de 4 checkpoints por requisição. O bloco fixo de ~6K reenviado a todo turno é o candidato perfeito.

Regra de ouro (prefix match): o cache casa por prefixo — qualquer byte alterado invalida o resto. Ordem de render: tools → system → messages. No Bedrock você marca um cachePoint explícito (ex.: AFTER_SYSTEM): tudo antes dele é cacheado — então ponha o estável primeiro (system congelado, tools determinísticas) e o volátil depois (a conversa). Nunca injete timestamp/UUID/"data de hoje" no system prompt — invalida o cache a cada requisição.

5 · A tensão: caching vs. sumarização

★ Insight — Sumarizar e cachear puxam em direções opostas. A janela (mensagens/tokens) é append-only: você adiciona no fim, o prefixo fica byte-a-byte igual → o cache continua válido (~0,1×). Já a sumarização reescreve o meio (10 turnos → 1 resumo) → muda o prefixo → invalida o cache dali pra frente. Prática para o banco: janela + caching no caso comum; sumarize com parcimônia, só quando o custo da history crua superar o de quebrar o cache. Medir, não adivinhar.

6 · A API verificada (LangChain4j 1.16.2)

ChatMemory (dev.langchain4j.memory) tem add, messages() e clear(). Duas janelas (dev.langchain4j.memory.chat):

  • MessageWindowChatMemory.withMaxMessages(int) — últimas N mensagens.
  • TokenWindowChatMemory.withMaxTokens(int, TokenCountEstimator) — últimos N tokens. Prefira esta no orçamento de 200K (o despejo acompanha o custo real em tokens).

Despejo é mais antigo primeiro e indivisível. Dois comportamentos automáticos que evitam bugs sérios:

  • A SystemMessage é fixada e nunca despejada — guardrails do banco sobrevivem (e é a mesma SystemMessage que você vai cachear: pinning + caching se reforçam).
  • Resultado de tool órfão é despejado junto da sua AiMessage — evita o erro "resultado de tool sem o pedido".

★ Ponto de atenção verificado (vazamento de memória) — O AiServices retém TODA ChatMemory que já criou. Cada cliente que conversa uma vez deixa a memória presa para sempre. A correção: a interface estende ChatMemoryAccess (getChatMemory, evictChatMemory(memoryId)) — chame ao fim da conversa. Bônus: esse método é a sua ferramenta para o direito ao esquecimento (LGPD).

Contagem de tokens no Bedrock: estime local, confira remoto

O langchain4j-bedrock 1.16.2 não traz contador de tokens (verificado). A AWS tem a operação CountTokens (IAM bedrock:CountTokens), mas é remota — cara para chamar a cada add na memória. Logo, para a TokenWindowChatMemory ao vivo use um TokenCountEstimator local/aproximado e reserve o CountTokens da AWS para verificação ocasional. Não há tokenizador offline do Claude; um estimador local subconta ~15–20% — deixe folga no maxTokens. O consumo real (incl. cache) volta na resposta: faça cast de chatResponse.metadata().tokenUsage() para BedrockTokenUsage e leia cacheReadInputTokens().

Prompt caching no langchain4j-bedrock — a resposta de produção

Armadilha de provedor (verificada): cacheSystemMessages(true)/cacheTools(true) são métodos reais, mas do módulo langchain4j-anthropic (API direta) — no caminho Bedrock eles não existem e têm efeito ZERO. Compila e não cacheia nada. É o erro "parece certo" que só a fonte pega.

O jeito Bedrock (verificado em BedrockChatRequestParameters @ 1.16.2): um cachePoint via defaultRequestParameters:

BedrockChatModel model = BedrockChatModel.builder()
    .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0")   // inference profile cross-region
    .region(Region.US_EAST_1)
    .defaultRequestParameters(BedrockChatRequestParameters.builder()
        .promptCaching(BedrockCachePointPlacement.AFTER_SYSTEM, // cacheia tools+system (o prefixo fixo)
                       CacheTTL.VALUE_5_M)                       // VALUE_5_M / VALUE_1_H (enum do AWS SDK)
        .build())
    .build();

TTL por modelo (verificado, docs AWS): o TTL padrão de 5 min (CacheTTL.VALUE_5_M) é universal; o TTL estendido de 1 h (CacheTTL.VALUE_1_H) existe só em Claude Opus 4.5, Haiku 4.5 e Sonnet 4.5 — não em Sonnet 4.6 nem Opus 4.6 (5 min apenas). Escolher VALUE_1_H num modelo sem suporte não estende o cache.

Confirme via BedrockTokenUsage.cacheReadInputTokens() (após cast do tokenUsage()) — se ficar em zero, o prefixo não atingiu o mínimo do modelo (4.096 tok no Sonnet 4.5) ou há um invalidador silencioso. Use AFTER_SYSTEM com uma SystemMessage do core: a janela de memória não trata a BedrockSystemMessage granular, então o cachePoint por placement é o caminho confiável com memória.

7 · O assistente do banco: memória + caching + privacidade

BedrockChatModel model = BedrockChatModel.builder()
    .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0")
    .region(Region.US_EAST_1)
    .defaultRequestParameters(BedrockChatRequestParameters.builder()
        .promptCaching(BedrockCachePointPlacement.AFTER_SYSTEM, CacheTTL.VALUE_5_M)
        .build())
    .build();

interface AssistenteBanco extends ChatMemoryAccess {           // poder de despejar/esquecer
    @SystemMessage("You are the assistant of Bank X. Use tools for account data; "
        + "never reveal another customer's data. Answer in Brazilian Portuguese.")
    String conversar(@MemoryId String clienteId, @UserMessage String pergunta);
}

AssistenteBanco assistente = AiServices.builder(AssistenteBanco.class)
    .chatModel(model)
    .tools(new FerramentasBanco(contaService))
    .chatMemoryProvider(clienteId -> TokenWindowChatMemory.builder()
        .id(clienteId)
        .maxTokens(20_000, estimadorLocalBarato)   // bem abaixo de 200K; estimador LOCAL (não o remoto)
        .chatMemoryStore(redisStore)               // durável por clienteId (PII em repouso!)
        .build())
    .build();

// ... ao encerrar a conversa (ou sob pedido de LGPD):
assistente.evictChatMemory(clienteId);             // libera memória e/ou "esquece" o cliente

Cada peça resolve um problema da estratégia: TokenWindowChatMemory(20_000) controla o N² e cabe folgado em 200K; o promptCaching(AFTER_SYSTEM) do Bedrock derruba o custo do prefixo fixo; redisStore dá durabilidade (ao custo de PII em repouso — Lição 13); evictChatMemory resolve vazamento e direito ao esquecimento.

8 · Pratique

Quiz — recupere de memória, sem reler

1. Num orçamento de 200K, qual o problema real da gestão de contexto?

Caber raramente é o gargalo; o custo recorrente (prefixo + N²) é.

2. Qual ajuste do BedrockChatModel ataca diretamente o custo N²?

cacheSystemMessages é do langchain4j-anthropic (no-op no Bedrock). No Bedrock, o cachePoint AFTER_SYSTEM faz a leitura do prefixo custar muito menos a cada turno.

3. Como contar tokens da janela ao vivo no Claude-via-Bedrock?

Não há tokenizador offline do Claude e o langchain4j-bedrock não conta tokens; o CountTokens da AWS é remoto. Estime local na janela, confira remoto de vez em quando.
