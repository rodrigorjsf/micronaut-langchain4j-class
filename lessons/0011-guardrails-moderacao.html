Guardrails e moderação: defendendo a saída

A Lição 10 deixou um objeto tipado, mas não-confiável. Aqui está a rede de defesa — e a separação cuidadosa de dois sistemas que só compartilham a palavra "guardrail", sob uma fronteira inegociável: guardrail ≠ autorização.

Por que isto importa para a sua missão — num banco, a saída do modelo pode vazar PII (Personally Identifiable Information / Informação Pessoalmente Identificável), alucinar um valor ou obedecer a uma injeção. O ganho desta lição: defender essa saída com um OutputGuardrail que reprompta + o Bedrock Guardrails como rede da plataforma — sabendo que isso é defesa probabilística, e que a trava de acesso continua determinística.

2 · LangChain4j: Input e Output guardrails (verificado)

  • InputGuardrail — validate(UserMessage); helpers default success(), failure(String), fatal(String). Sem retry/reprompt: a 1ª falha encerra a cadeia e vira GuardrailException. Para: pegar injeção/abuso antes do modelo.
  • OutputGuardrail — validate(AiMessage); além de success(), tem reprompt(msg, reprompt) (re-invoca com novo prompt) e retry(msg) (re-invoca com o mesmo).

★ Armadilha verificada — reprompt(...)/retry(...) são métodos default da INTERFACE OutputGuardrail, não estáticos de OutputGuardrailResult. OutputGuardrailResult.reprompt(...) não compila; e o estático OutputGuardrailResult.failure(...) recebe List<Failure>, não String. Chame reprompt(...) direto dentro do validate(...).

Registro: @InputGuardrails({...}) e @OutputGuardrails(value={...}, maxRetries=2) (pacote dev.langchain4j.service.guardrail). Precedência: método > classe > builder. maxRetries é o total de invocações do modelo — a chamada inicial conta como a 1ª tentativa (verificado no OutputGuardrailExecutor 1.16.2: maxAttempts = maxRetries, sem +1): default 2 → até 2 invocações (inicial + 1 reprompt); 0→1; negativo→trata como 2; esgotou → OutputGuardrailException. A entrada não tem maxRetries.

class PromptInjectionGuard implements InputGuardrail {
    @Override public InputGuardrailResult validate(UserMessage um) {
        String t = um.singleText().toLowerCase();
        if (t.contains("ignore as instrucoes") || t.contains("ignore previous"))
            return failure("Possivel prompt injection");   // helper default; sem retry na entrada
        return success();
    }
}

class PiiPtBrGuard implements OutputGuardrail {
    @Override public OutputGuardrailResult validate(AiMessage resp) {
        String txt = resp.text();
        if (txt == null || txt.isBlank())
            return reprompt("Resposta vazia", "Responda em pt-BR, em JSON valido."); // re-invoca c/ novo prompt
        if (vazaContaDeOutroCliente(txt))
            return reprompt("Possivel vazamento de PII", "NAO inclua dados de outro cliente.");
        return success();
    }   // reprompt/retry: metodos DEFAULT da interface — nunca OutputGuardrailResult.reprompt(...)
}

interface BankAssistant {
    @InputGuardrails(PromptInjectionGuard.class)
    @OutputGuardrails(value = PiiPtBrGuard.class, maxRetries = 2)  // 2 = total de invocações (inicial + 1 reprompt)
    Triagem triar(@V("msg") String msg);
}

3 · AWS Bedrock Guardrails (a rede da plataforma)

Sistema separado, do provedor. Políticas (docs AWS): filtros de conteúdo (incl. Prompt Attack = jailbreak/injection), tópicos negados, filtros de palavra, informação sensível (PII: bloquear ou mascarar) e contextual grounding check (anti-alucinação, p/ RAG — Retrieval-Augmented Generation / Geração Aumentada por Recuperação).

BedrockGuardrailConfiguration gc = BedrockGuardrailConfiguration.builder()
    .guardrailIdentifier("gr-xxxxxxxx")
    .guardrailVersion("DRAFT")
    .streamProcessingMode(BedrockGuardrailConfiguration.ProcessingMode.SYNC) // SYNC = veredito antes do token
    .build();   // 3 campos; NAO ha 'trace' (langchain4j fixa GuardrailTrace.ENABLED)

BedrockChatRequestParameters params = BedrockChatRequestParameters.builder()
    .guardrailConfiguration(gc)   // builder = guardrailConfiguration(...); getter = bedrockGuardrailConfiguration()
    .build();

In-band (no Converse) exige IAM (Identity and Access Management / Gestão de Identidade e Acesso) bedrock:InvokeModel; out-of-band (pré-triagem) usa ApplyGuardrail (bedrock:ApplyGuardrail). O GuardrailAssessment traz action (ANONYMIZED|BLOCKED|NONE|UNKNOWN) e policy — flags de avaliação, não um veredito de permissão.

4 · Moderação: para Claude-on-Bedrock, Guardrails É a moderação

O langchain4j-core tem a SPI (Service Provider Interface / Interface de Provedor de Serviço) ModerationModel, mas as implementações em 1.16.2 são só OpenAI, Mistral e Watsonx — não há ModerationModel para Bedrock/Anthropic. Logo, neste banco não existe moderate(); a moderação é o Bedrock Guardrails (in-band ou via ApplyGuardrail).

5 · A fronteira que ancora tudo: guardrail ≠ autorização

★ Insight — Guardrails (as duas camadas) são defesa probabilística, NUNCA a camada de autorização. Onde a API mostra isso: o Javadoc do @InputGuardrails diz "It does not replace a moderation model"; o GuardrailAssessment expõe flags (BLOCKED/ANONYMIZED), não allow/deny; e o ProcessingMode.ASYNC libera tokens antes de terminar a avaliação — prova de que é best-effort. A autorização real é IAM (bedrock:InvokeModel, condição bedrock:GuardrailIdentifier) + a sua autorização de negócio no código (Lição 1; filtro forçado pelo servidor da Lição 9). Guardrail reduz risco residual; não substitui controle de acesso.

Nota de compliance: conteúdo bloqueado aparece em texto puro nos Bedrock Model Invocation Logs, se ligados — atenção a PII em pt-BR. 
entrada → [InputGuardrail (app)] → modelo  ⟵ [Bedrock Guardrails (provedor)]
        → [OutputGuardrail (app, pode repromptar)] → resposta
        — e a AUTORIZACAO (IAM + codigo deterministico) por fora de tudo

6 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. O que distingue as duas famílias de "guardrail"?

Compartilham só a palavra. Uma é código in-process (que você escreve); a outra é config gerenciada na AWS, aplicada à chamada.

2. Um OutputGuardrail bloqueou um vazamento de PII. Isso é a sua autorização?

Guardrail é best-effort (ASYNC chega a liberar tokens antes de avaliar). Autorização = IAM + sua lógica determinística. Guardrail só reduz risco residual.

3. Como sinalizar "resposta ruim, tente de novo" num OutputGuardrail?

reprompt/retry são métodos default da interface OutputGuardrail. O estático em OutputGuardrailResult não existe (e failure recebe List<Failure>, não String).
