RAG no LangChain4j: um store, dois mundos sem vazar

A Lição 5 terminou com um esboço da arquitetura RAG do banco. Aqui ele vira código verificado do LangChain4j 1.16.2, invocando o Claude via AWS Bedrock — com a peça crítica sendo o filtro forçado pelo servidor que separa a conta nacional da global.

Por que isto importa para a sua missão — o desafio central do seu banco é manter conta nacional e conta global (regras de backends distintos) jamais misturadas. Esta lição entrega um ganho: cablar um ContentRetriever com filtro de entitulamento forçado pelo servidor no seu AiService, de modo que um cliente nacional nunca recupere um documento global.

1 · Os dois níveis de API (o mesmo padrão da Lição 6)

Assim como o AiServices embrulha o ChatModel (Lição 6), o RAG entra no AiServices por dois caminhos — e não há um terceiro:

CaminhoMétodo no builderQuando
RAG ingênuo.contentRetriever(ContentRetriever)caso comum: "busque top-k e injete"
RAG avançado.retrievalAugmentor(RetrievalAugmentor)transformação/roteamento/re-rank/citação (§7)
Correção verificada na fonte (1.16.2): não existe AiServices.retriever(...) — código que o chama não compila (a palavra "retriever" só sobrevive na string de erro). Os dois métodos são mutuamente exclusivos: setar ambos lança "Only one out of [retriever, contentRetriever, retrievalAugmentor] can be set". O caminho simples monta, por baixo, um DefaultRetrievalAugmentor.

As interfaces (ambas em langchain4j-core): ContentRetriever tem List<Content> retrieve(Query) (lista ordenada por relevância); RetrievalAugmentor tem AugmentationResult augment(AugmentationRequest) — o ponto de entrada do fluxo.

2 · Os dois pipelines da Lição 5, agora em classes

INGESTÃO (offline)   Documento → Quebrar → Embeddar → Guardar
                                  DocumentSplitter · EmbeddingModel · EmbeddingStore
                                  └──────── EmbeddingStoreIngestor ────────┘

RECUPERAÇÃO (online) Pergunta → embeddar → buscar top-k → FILTRAR → injetar
                                  └─ EmbeddingStoreContentRetriever ─┘
Pegadinha de módulos (verificada): o rag.*, o Filter, o EmbeddingStoreContentRetriever e o EmbeddingStoreIngestor vivem em langchain4j-core; mas o AiServices, o DocumentSplitters e o InMemoryEmbeddingStore vivem no módulo agregador langchain4j. E há dois trilhos de versão na tag 1.16.2: stable 1.16.2 (core, langchain4j, bedrock) vs beta 1.16.2-beta26 (pgvector, cohere, voyage). Misturar a versão errada não resolve → gerencie tudo pelo langchain4j-bom.

3 · O embedding na Bedrock (a escolha do banco, para pt-BR)

O modelo de embedding é separado do de chat (Lição 5 §3). A Anthropic não tem API de embeddings (verificado: langchain4j-anthropic só traz chat) — e como a invocação do banco é via Bedrock, o embedding também vem da Bedrock:

  • Produção (pt-BR): BedrockCohereEmbeddingModel + cohere.embed-multilingual-v3 (1024-dim, GA, feito para 100+ idiomas incl. português). O amazon.titan-embed-text-v2:0 é GA (desde abr/2024), mas é otimizado para inglês (não multilíngue-first) — por isso, para pt-BR, prefira o Cohere multilíngue como caminho primário.
  • Pegadinha crítica (verificada): no Cohere o inputType é obrigatório e fixo por instância. A AWS manda embeddar o corpus com SEARCH_DOCUMENT e a pergunta com SEARCH_QUERY → você precisa de DUAS instâncias (uma para o ingestor, outra para o retriever). Cada texto é limitado a 512 tokens → quebre pequeno no splitter.

★ Insight — Assimetria de credenciais (verificada): os builders de embedding da Bedrock têm credentialsProvider(...), mas o BedrockChatModel não tem — ele fixa DefaultCredentialsProvider.create(). Para usar assume-role/STS no chat, você constrói seu próprio BedrockRuntimeClient e injeta via .client(...). É o tipo de detalhe que só a fonte revela — e que num banco (onde credenciais são auditadas) muda o desenho.

4 · A joia da coroa: TRÊS metadados e o filtro do servidor

O desafio nacional-vs-global resolve-se com filtro por metadado. Mas há três Metadata confundíveis — e o erro mora em trocá-las:

TipoPapelTem account_scope?
data.document.Metadatatags de negócio no segmentoSIM — é o que Filter.test() lê
rag.query.Metadatacontexto da query (chatMemoryId, …)não
Content.metadata()bookkeeping (enum ContentMetadata)não
A linha mais sensível à segurança (corrigida pela fonte): query.metadata().get("account_scope") NÃO EXISTE — a rag.query.Metadata não tem get(String). O caminho correto, com null-check (a Query de um argumento tem metadata()==null): 
String scope = query.metadata().invocationParameters().get("account_scope"); // <T> T get(String)
// ou, quando o escopo deriva do usuário logado:
Object userId = query.metadata().chatMemoryId(); // "Can be used to distinguish between users."

★ Insight — O escopo vem do SERVIDOR, não do modelo. Entitulamento é autorização (Lição 1: identidade na camada determinística). O dynamicFilter (Function<Query, Filter>) lê o escopo confiável passado pelo servidor (em invocationParameters, ou derivado do chatMemoryId) — nunca de algo que o modelo escreveu. Ele se compõe com AND a uma faceta de produto que o modelo pode sugerir (self-query, Lição 5 §8): Filter.and(escopoDoServidor, produtoDoModelo). O modelo ajuda a refinar; ele jamais decide a fronteira de acesso.

A garantia de separação (verificada): IsEqualTo.test() retorna false quando o segmento não contém a chave (guarda containsKey). Logo um segmento "global" nunca casa metadataKey("account_scope").isEqualTo("national"). É esse false que mantém os dois mundos apartados.

Outras armadilhas verificadas: filter(null) é um no-op silencioso (não limpa um filtro já setado); account_scope tem de ser String (os tipos da data.document.Metadata são um conjunto fechado); e minScore/maxResults no builder do retriever são validados na primeira query, não no build().

5 · O portão de segurança: nem todo store APLICA o filtro

EmbeddingStore.search(...) avisa, verbatim: "not all EmbeddingStore implementations support Filtering." Um store que ignora o filtro transforma seu entitulamento em no-op silencioso = vazamento entre contas, sem nenhum erro. Por isso a escolha do store é decisão de segurança, não só de infraestrutura.
  • InMemoryEmbeddingStore (módulo langchain4j; sem builder() → use new InMemoryEmbeddingStore<>()): APLICA o filtro em processo — o search() faz if (!filter.test(meta)) continue;. Ótimo para o demo.
  • PgVectorEmbeddingStore (langchain4j-pgvector, beta26; cosine-only/ivfflat-only): APLICA compilando o Filter para SQL. MetadataStorageMode default = COMBINED_JSON. Caminho de produção.

6 · O assistente do banco (verificado na fonte)

static final String SCOPE = "account_scope";   // valor SEMPRE String ("national"/"global")

// (0) Pontas Bedrock — DUAS instâncias de embedding (inputType fixo por instância).
EmbeddingModel docEmbedder = BedrockCohereEmbeddingModel.builder()
        .model(Model.COHERE_EMBED_MULTILINGUAL_V3)  // "cohere.embed-multilingual-v3" (pt-BR)
        .inputType(InputType.SEARCH_DOCUMENT)       // OBRIGATÓRIO — indexar o corpus
        .region(Region.US_EAST_1).build();
EmbeddingModel queryEmbedder = BedrockCohereEmbeddingModel.builder()
        .model(Model.COHERE_EMBED_MULTILINGUAL_V3)
        .inputType(InputType.SEARCH_QUERY)          // OBRIGATÓRIO — embeddar a pergunta
        .region(Region.US_EAST_1).build();
ChatModel chat = BedrockChatModel.builder()
        .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0") // inference profile cross-region
        .region(Region.US_EAST_1).build();
EmbeddingStore<TextSegment> store = new InMemoryEmbeddingStore<>(); // demo (aplica o filtro)

// (1) INGESTÃO — tag o discriminador UMA vez no Document; o splitter o copia
//     verbatim para CADA segmento. (Parágrafo→Linha→Frase→Palavra→Caractere.)
void ingest(String texto, String scope) {
    Document doc = Document.from(texto, Metadata.from(SCOPE, scope));
    EmbeddingStoreIngestor.builder()
            .documentSplitter(DocumentSplitters.recursive(300, 30)) // pequeno (limite 512 tok do Cohere)
            .embeddingModel(docEmbedder).embeddingStore(store).build()
            .ingest(doc);                            // IngestionResult só tem tokenUsage()
}

// (2) RETRIEVAL — filtro POR USUÁRIO, forçado pelo servidor.
Function<Query, Filter> escopoDoUsuario = query -> {
    String scope = null;
    if (query.metadata() != null && query.metadata().invocationParameters() != null)
        scope = query.metadata().invocationParameters().get(SCOPE);
    if (scope == null) scope = "national";           // default seguro: nunca vaza global
    return metadataKey(SCOPE).isEqualTo(scope);      // segmento sem a chave nunca casa
};
EmbeddingStoreContentRetriever retriever = EmbeddingStoreContentRetriever.builder()
        .embeddingStore(store).embeddingModel(queryEmbedder)   // o embedder de QUERY
        .maxResults(5).minScore(0.6)
        .dynamicFilter(escopoDoUsuario).build();

// (3) WIRING — caminho SIMPLES (não existe .retriever(...)).
interface BankAssistant { String chat(String pergunta); }
BankAssistant assistant = AiServices.builder(BankAssistant.class)
        .chatModel(chat)
        .contentRetriever(retriever)                 // RAG ingênuo
        .build();

Uma EmbeddingStore, account_scope no metadado do segmento, um dynamicFilter que lê o escopo confiável do servidor, e um store que aplica o Filter.test. O BedrockChatModel cabla como qualquer ChatModel. Resultado: o cliente nacional nunca recupera o doc global.

7 · Prévia: o RAG avançado mapeado na Lição 5 §8

Quando o ingênuo não basta, o DefaultRetrievalAugmentor destrincha um pipeline de 5 estágios — cada um é o nome-de-framework de um padrão que a Lição 5 §8 já ensinou:

EstágioClasse(s) 1.16.2Padrão (Lição 5 §8)
1 · TransformarQueryTransformer → Compressing/Expandingmulti-query, HyDE
2 · RotearQueryRouter → Default / LanguageModelself-query / roteamento
3 · AgregarContentAggregator → Default(RRF) / ReRankingre-ranqueamento
4 · InjetarContentInjector → Defaultinjeção + citação

Notas verificadas: o queryRouter é obrigatório (NPE no build() se nulo); DefaultQueryRouter manda toda query a todos os retrievers (para escolher por backend, use LanguageModelQueryRouter); as classes LLM usam ChatModel (não o legado ChatLanguageModel). Para "um store + filtro", o caminho do §6 é mais simples e mais seguro. Mergulho profundo numa lição futura.

8 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Onde mora a tag account_scope que o filtro inspeciona?

A rag.query.Metadata nem tem get(String); a Content.metadata() é só bookkeeping. O filtro lê o metadado do segmento.

2. Por que o escopo (nacional/global) vem do servidor, não do modelo?

Identidade/autorização vivem na camada determinística (Lição 1). O dynamicFilter lê o escopo confiável de invocationParameters/chatMemoryId.

3. E se o EmbeddingStore não suportar filtro de metadado?

"not all EmbeddingStore implementations support Filtering." Sem suporte, o filtro é no-op silencioso. InMemory e pgvector aplicam; confirme o seu.
