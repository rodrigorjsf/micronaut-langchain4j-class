Saída estruturada: do texto livre ao objeto tipado

O agente já busca (tools, RAG) e lembra (memory). Agora a forma da saída: como fazer o AiService devolver um objeto tipado que o backend do banco pode rotear e decidir — e por que ele é tipado, mas ainda não-confiável.

Por que isto importa para a sua missão — o backend agêntico precisa decidir ("isto é um Pix ou uma dúvida de tarifa?", "o valor é R$ 100,00?"). Decisão pede dado tipado, não uma frase. Esta lição entrega um ganho: uma triagem em record que o seu código roteia — com a fronteira clara de que tipado ≠ confiável.

1 · Texto livre vs. objeto tipado

Texto livre serve para mostrar ao cliente. Mas o backend precisa decidir e persistir — e isso pede uma saída que o código consiga inspecionar com segurança de tipos.

Você quer…Forma da saída
Exibir uma resposta ao clienteString (texto livre)
Rotear / decidir / persistir no backendPOJO/record tipado (saída estruturada)

2 · O tipo de retorno é o contrato

No AiServices 1.16.2, o tipo de retorno do método dirige tudo (verificado em DefaultAiServices). Não há anotação @Json nem responseFormat por chamada.

  • Suportados: boolean/int/long/double, BigDecimal, LocalDate/LocalDateTime, enum, List<T>/Set<T> e, por padrão, POJO/record.
  • @Description (FQN dev.langchain4j.model.output.structured.Description, @Target({FIELD, TYPE})) descreve campos/classe para o schema — não mira parâmetros de método.
  • Result<T> embrulha o tipo com content(), tokenUsage(), sources() (os chunks do RAG — Lição 9), finishReason(). Dirige o mesmo schema do T.
Armadilha verificada: Map é degenerado — não há MapOutputParser, e schemaNotRequired(Map.class) pula schema e prompt. Use um POJO/record concreto.

3 · Os dois caminhos do AiServices (e só dois)

O DefaultAiServices escolhe entre exatamente duas estratégias. (Não há um terceiro caminho via tool-use para saída por tipo de retorno — tool-use é function calling, outro mecanismo.)

CaminhoQuandoO que faz
(A) JSON schema nativosupportsJsonSchema() & !streaming & há schemamonta ResponseFormat.type(JSON).jsonSchema(...) — restringe a geração
(B) JSON por promptcapability ausente ou sem schemaem silêncio anexa uma instrução textual — zero garantia

supportsJsonSchema() = chatModel.supportedCapabilities().contains(RESPONSE_FORMAT_JSON_SCHEMA). O ResponseFormatType tem só TEXT e JSON — "schema nativo" é type(JSON) + um JsonSchema não-nulo.

4 · Bedrock: o opt-in obrigatório (e o fallback silencioso)

★ Insight — O gêmeo do cacheSystemMessages da Lição 8. Por padrão, o BedrockChatModel não anuncia a capability RESPONSE_FORMAT_JSON_SCHEMA (copy(null) = Set.of()). Resultado: o AiServices cai no fallback por prompt em silêncio — sem warning, sem exceção. Você acha que tem schema forçado; tem só uma "sugestão" de formato. De novo: compila e não faz o que você acha — só a fonte primária pega.

A correção — opt-in explícito:

ChatModel model = BedrockChatModel.builder()
    .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0")
    .region(Region.US_EAST_1)
    .supportedCapabilities(Capability.RESPONSE_FORMAT_JSON_SCHEMA)  // sem isto: fallback silencioso
    .build();

Com o opt-in, o caminho nativo do Converse monta OutputConfig.textFormat(OutputFormat.type(JSON_SCHEMA)...) — separado do .guardrailConfig(...) (saída e guardrails são ortogonais; guardrails são a Lição 11). Pedir JSON sem schema lança UnsupportedFeatureException. Comportamento provado pelo teste de integração BedrockAiServiceWithJsonSchemaIT (com us.anthropic.claude-haiku-4-5-20251001-v1:0).

5 · O assistente do banco: triagem tipada

enum Intencao { CONSULTA, PIX, PAGAMENTO_BOLETO, FORA_DE_ESCOPO }

record Triagem(
        Intencao intencao,
        @Description("produto citado, ex.: CDB, cartao, Pix; senao null") String produto,
        @Description("valor em reais se houver; senao null")              BigDecimal valor,
        @Description("resumo em pt-BR da intencao do cliente")            String resumo
) {}

interface Triador {
    @UserMessage("Classifique a mensagem do cliente do banco: {{msg}}")
    Triagem triar(@V("msg") String msg);

    // Variante com metadados (tokenUsage, sources do RAG, finishReason):
    @UserMessage("Classifique: {{msg}}")
    Result<Triagem> triarComMetadados(@V("msg") String msg);
}

ChatModel model = BedrockChatModel.builder()
        .modelId("us.anthropic.claude-sonnet-4-5-20250929-v1:0")
        .region(Region.US_EAST_1)
        .supportedCapabilities(Capability.RESPONSE_FORMAT_JSON_SCHEMA) // opt-in!
        .build();

Triador triador = AiServices.create(Triador.class, model);
Triagem t = triador.triar(mensagemDoCliente);

// t é TIPADO, mas NÃO-CONFIÁVEL. NUNCA aja só porque o modelo classificou "PIX":
if (t.intencao() == Intencao.PIX) {
    // valide o valor, confirme com o cliente e AUTORIZE na camada determinística
    // (Lição 1 + Lição 11). O tipo garante a FORMA, não a VERDADE nem a permissão.
}

6 · Tipado ≠ confiável (a ponte para a Lição 11)

★ Insight — Saída estruturada garante a FORMA, nunca a VERDADE — nem a autorização. O schema faz o JSON casar com o record; ele não garante que valor é o correto, que intencao é honesta, ou que o cliente pode fazer aquilo. O caminho nativo restringe a geração, mas ainda é probabilístico; o fallback por prompt não restringe nada. Trate o POJO parseado como entrada não-confiável: valide regras de negócio e autorize no seu código (determinístico) antes de agir. É o princípio da Lição 1 — e o gancho da Lição 11 (guardrails + moderação), a rede de defesa sobre esta saída.

7 · Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. O que define o formato da saída estruturada no AiServices?

O DefaultAiServices lê o returnType do método e gera o schema/parser a partir dele — não há @Json nem responseFormat por chamada.

2. No Bedrock, sem o opt-in da capability de JSON schema, o que ocorre?

supportedCapabilities default = Set.of() → fallback silencioso. Opte por .supportedCapabilities(RESPONSE_FORMAT_JSON_SCHEMA) para o schema nativo.

3. Recebeu um record tipado e válido do modelo. Pode confiar nele?

O tipo garante FORMA, não VERDADE nem permissão. E nem guardrail é autorização (Lição 1/11): valide e autorize na camada determinística.
