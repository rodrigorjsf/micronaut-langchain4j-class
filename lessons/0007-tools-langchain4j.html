Parte 2 · LangChain4j · Lição 7

Tools (@Tool): design e API verificada (1.16.2)

A tool é a "lógica de decisão" do seu agente — e o leitor dela é o modelo, não um humano. Aqui a tese vira regras de design (Parte A) e desce ao código real do LangChain4j, símbolo por símbolo conferido contra a tag 1.16.2 (Parte B).

Por que isto importa para a sua missão — você está construindo um assistente bancário agêntico em produção sobre LangChain4j + Micronaut 5 + Java 25. Cada tool move (ou expõe) o dinheiro do cliente. Granularidade errada custa tokens e degrada decisões; descrição imprecisa faz o agente chamar a tool errada; e o comportamento-padrão de erro do LangChain4j vaza PII para o modelo se você não o sobrescrever. Esta lição é onde o desenho seguro encontra a API verificada.

Parte A · 1 — Granularidade: nem grossa demais, nem fina demais

A pergunta de design nº 1: o que é "uma" ferramenta? Pense numa loja: um único balcão que vende tudo (eletrônicos, comida, roupa) confunde o cliente; vinte balcões com um produto cada o esgotam. O equilíbrio é o departamento — uma capacidade coerente.

  • Grossa demais (coarse): getCartaoCompleto() que devolve fatura + limite + lançamentos + benefícios num único JSON gigante. Cada chamada despeja milhares de tokens na mesa (custo/latência), a maior parte irrelevante, e ainda piora a resposta por "ruído" (perdido no meio).
  • Fina demais (fine): getLimiteDisponivel(), getLimiteTotal(), getLimiteUtilizado(), getDataFechamento()… vinte micro-tools. Vinte descrições na mesa toda chamada, mais decisões para o modelo errar, mais viagens sequenciais (latência).

A medida certa (Goldilocks): uma tool = uma capacidade que faz sentido para o usuário. getFaturaCartao(), getLimiteCartao(), getLancamentosCartao(periodo).

★ Insight — Desenhe tools ao redor das INTENÇÕES do usuário, não da forma da sua API de backend. É tentador espelhar 1:1 os endpoints REST que já existem no banco — mas o modelo não pensa em endpoints, pensa em intenções ("quero ver minha fatura"). Uma tool que combina duas chamadas de backend numa capacidade limpa, ou que divide um endpoint gordo em duas tools focadas, quase sempre melhora a decisão do agente. A camada de tools é uma fachada (facade) projetada para o modelo, não um espelho do seu backend.

Parte A · 2 — Nomes: verbo + substantivo, inequívocos e distintos

O nome é a primeira coisa que o modelo lê. Regras:

  • Verbo + substantivo: getFaturaCartao, agendarPagamentoBoleto, fazerPix.
  • Inequívoco e distinto: se duas tools têm nomes/descrições parecidos, o modelo confunde. getSaldoContaNacional vs. getSaldoContaGlobal precisam ser claramente distintos.

Surge aqui uma decisão de arquitetura recorrente no nosso banco (nacional vs. global):

AbordagemExemploPrósContras
Duas tools distintas getSaldoNacional(), getSaldoGlobal() Decisão explícita; descrições podem divergir; roteamento de backend trivial Dobra a contagem de tools (×2 para cada produto → "muitas tools", seção 7)
Uma tool parametrizada getSaldo(tipoConta: NACIONAL|GLOBAL) Menos tools na mesa; uma descrição O modelo precisa preencher o parâmetro certo; risco de pôr o errado

★ Regra prática — quando os produtos têm regras e backends realmente diferentes (caso do nacional vs. global), tools distintas costumam dar decisões mais confiáveis; quando a diferença é só um filtro, a tool parametrizada com enum (que restringe os valores possíveis) é mais enxuta. Decida produto a produto.

Parte A · 3 — Descrições: você está escrevendo para o modelo

A descrição é o que o modelo usa para decidir se e quando chamar. Uma boa descrição diz:

  1. O que a tool faz.
  2. Quando usá-la (e, crucialmente, quando NÃO).
  3. Unidades e formatos (moeda, datas em ISO, fuso).
  4. Efeitos colaterais (se move dinheiro/estado).
RUIM:  "Retorna fatura."
BOM:   "Retorna a fatura ATUAL (em aberto) do cartão de crédito do cliente
        autenticado, com valor em BRL e data de vencimento (ISO-8601).
        Use para perguntas sobre o valor/vencimento da fatura corrente.
        NÃO use para faturas de meses anteriores — para isso, use
        getFaturaCartao(mesReferencia)."

★ Insight — No mundo real, ~80% do "tuning" de um agente é reescrever descrições de tools — não trocar o modelo nem mexer em hiperparâmetros. Quando o agente chama a tool errada, traz a conta errada ou deixa de chamar quando deveria, a primeira (e quase sempre a única) correção necessária é tornar as descrições mais precisas e mutuamente exclusivas. Trate suas descrições de tools como código de produção versionado e testado, porque é exatamente o que elas são: a "lógica de decisão" do seu agente vive nelas.

Parte A · 4 — Parâmetros: mínimos, tipados, e sem identidade

  • Sem parâmetros de identidade/segurança. getFaturaCartao não recebe clienteId — o servidor o injeta da sessão autenticada. O modelo só fornece intenção (qual mês), nunca quem é o cliente.
  • Mínimos. Cada parâmetro é uma chance de o modelo alucinar um valor. Menos é mais.
  • Tipados e restritos. Use enums (tipoConta: NACIONAL|GLOBAL) para fechar o espaço de valores; datas em ISO-8601; descreva cada parâmetro (na Parte B veremos a anotação que faz isso).
  • Validados antes de executar. Esquema + regras de negócio + autorização, sempre na sua camada determinística.

Parte A · 5 — Valores de retorno: pequenos, moldados, e com erro como dado

O retorno da tool vira tokens na mesa. Princípios:

  • Pequeno e moldado. Não devolva o JSON cru do backend (gigante, vaza campos internos e PII, custa tokens). Devolva só o necessário, já moldado: { valor: 2310.55, moeda: "BRL", vencimento: "2026-06-20", status: "ABERTA" }.
  • Estruturado e estável. Um formato consistente ajuda o modelo a interpretar.
  • Erro como dado, não como exceção que mata o loop. Se o backend global cai, a tool deve devolver algo como { erro: "BACKEND_GLOBAL_INDISPONIVEL", mensagem: "Conta global temporariamente indisponível." } — para o modelo reagir (pedir desculpas, sugerir tentar depois, escalar), em vez de o loop estourar. (Como o LangChain4j faz isso por baixo é tema da Parte B.)
⚠️ Minimização de dados é segurança. Quanto menos a tool devolve, menos PII (Personally Identifiable Information / Informação Pessoalmente Identificável) entra no contexto do LLM (e em logs/traces). "Devolver só o necessário" não é só economia — é privacidade.

Parte A · 6 — Tools de leitura vs. de ação

Separe e sinalize:

  • Leitura (getSaldo, getFatura): idempotentes, seguras, podem ser executadas quando o modelo pede.
  • Ação (fazerPix, agendarPagamentoBoleto, contratarFinanciamento): movem dinheiro/estado → exigem confirmação explícita, idempotência e autorização reforçada.

Uma convenção de nomes que ajuda humanos e o modelo: verbos de consulta (get…, consultar…) vs. verbos de ação (fazer…, agendar…, contratar…).

Parte A · 7 — O problema das "muitas tools"

Nosso banco tem todos os produtos: boleto, DDA, Pix, Super Extrato, cartão/fatura, compras, shopping, financiamento, investimentos — × (nacional e global). Mapeado ingenuamente, isso são dezenas de tools, e cada descrição vai na mesa em toda chamada. Consequências: custo fixo alto e decisões piores (o modelo se confunde entre muitas opções parecidas).

Mitigações:

  • Agrupar por domínio e enxugar descrições.
  • Vários agentes especializados sob um orquestrador (um agente de cartões, um de investimentos, um de Pix) — cada um vê só as suas tools.
  • Seleção dinâmica de tools (tool retrieval / RAG-sobre-tools): recuperar, por similaridade com a pergunta, apenas as tools relevantes para expor naquele turno — RAG aplicado ao próprio catálogo de ferramentas.
  • Namespacing claro (cartao.getFatura, pix.enviar).

Parte B · 1 — As três anotações

Aterrada por verificação: todos os símbolos abaixo foram conferidos contra o código-fonte da tag 1.16.2 no GitHub (não da memória). Onde um fato vem só da documentação, está marcado.

import dev.langchain4j.agent.tool.Tool;          // @Tool   (no método)
import dev.langchain4j.agent.tool.P;             // @P      (no parâmetro)
import dev.langchain4j.agent.tool.ToolMemoryId;  // @ToolMemoryId (no parâmetro)
AnotaçãoAtributos verificados (1.16.2)Notas
@Tool String name() (padrão = nome do método); String[] value() (a DESCRIÇÃO, várias entradas juntadas em multilinha); e três @Experimental: returnBehavior() (padrão TO_LLM), searchBehavior(), metadata() @Target(METHOD)
@P description() / value() (aliases — use um), name(), boolean required() padrão true, defaultValue() @Target(PARAMETER)
@ToolMemoryId (sem atributos) injeta o memory id no parâmetro par com @MemoryId do AI Service
★ Pegadinha verificada (não erre isto) — @Tool.value() é a DESCRIÇÃO, não o nome — e é um String[], não String. O nome é @Tool.name(), que por padrão é o nome do método. Muita gente (e muitos tutoriais) trocam os dois. E todo parâmetro @P é obrigatório por padrão (required = true); para torná-lo opcional, use @P(required = false), um defaultValue, ou um Optional<T>.

Parte B · 2 — Tipos permitidos e como o retorno chega ao modelo

Parâmetros: primitivos, boxed (Integer…), enums, POJOs e records (inclusive aninhados), tipos polimórficos (sealed), List/Set/Map, Optional<T> (torna opcional), ou zero parâmetros. Parâmetros injetados pelo framework (@ToolMemoryId) não são preenchidos pelo modelo.

Retorno (verificado no javadoc de Tool.java): o método pode retornar qualquer tipo, inclusive void.

RetornoO que é enviado ao modelo
voida string literal "Success"
Stringenviada como está (sem conversão)
qualquer outro (record, POJO, Map, coleção…)serializado em JSON
multimodal (Image, Content…)enviado como conteúdo multimodal (só em alguns provedores — docs)

★ Insight — Isso confirma a regra da Parte A (seção 5): retorne um record enxuto (vira JSON pequeno e estável) em vez do JSON cru do backend.

Parte B · 3 — O assistente do banco com tools reais

class FerramentasBanco {

    private final ContaService contas;   // sua camada determinística (auth, roteamento de backend)

    @Tool("Returns the credit card invoice (amount in BRL, due date ISO-8601) for the "
        + "authenticated customer's CURRENT cycle. Use for the current invoice; "
        + "do NOT use for past months.")
    FaturaResumo getFaturaCartao(@ToolMemoryId String clienteId) {        // ← identidade INJETADA, não vinda do modelo
        return contas.faturaAtual(clienteId);                            //   retorna um record pequeno → vira JSON
    }

    @Tool("Returns the account balance for the authenticated customer.")
    SaldoResumo getSaldo(@ToolMemoryId String clienteId,
                         @P("Account scope to query") TipoConta tipoConta) {  // ← enum restringe os valores
        return contas.saldo(clienteId, tipoConta);
    }
}

record FaturaResumo(double valor, String moeda, String vencimento, String status) {}
record SaldoResumo(double saldo, String moeda) {}
enum TipoConta { NACIONAL, GLOBAL }

★ Insight — @ToolMemoryId é exatamente a "identidade vem do servidor, nunca do modelo". Declaramos conversar(@MemoryId String clienteId, ...). Esse clienteId — que você vincula à sessão autenticada — flui automaticamente para o @ToolMemoryId String clienteId da tool. O modelo nunca toca nesse valor; ele só escolhe tipoConta. Assim, o ataque "engana o modelo para passar o ID de outro cliente" não tem superfície: o ID não está nas mãos do modelo. Atenção honesta: a garantia vem de você fazer memoryId = identidade autenticada — o framework só transporta; a segurança é da sua decisão de binding.

Parte B · 4 — Tratamento de erros: o coração da robustez (e um risco de banco)

Aqui o LangChain4j tem comportamentos-padrão que você precisa conhecer e, num banco, quase sempre sobrescrever:

SituaçãoPadrão na linha 1.x (verificado)O que fazer no banco
@Tool lança exceção AiServices captura e envia e.getMessage() ao modelo (para ele se corrigir). Não propaga ao chamador. ⚠️ Vaza dado interno/PII para o modelo e a conversa. Sobrescreva com toolExecutionErrorHandler e devolva mensagem sanitizada; logue a causa real via o Throwable error recebido pelo handler.
Argumentos inválidos (JSON malformado, obrigatório faltando) Caminho separado; padrão é lançar/abortar Trate com toolArgumentsErrorHandler (retornar texto para o modelo tentar de novo, ou abortar).
Tool alucinada (nome inexistente) hallucinatedToolNameStrategy → padrão THROW_EXCEPTION (aborta) Forneça um Function que devolva ToolExecutionResultMessage.from(req, "...") para o modelo se recuperar.
Loop sem fim maxToolCallingRoundTrips(int) → padrão 100 Reduza para um teto sensato (ex.: 6–8) para limitar custo.

Montando o assistente com essas travas (API verificada):

AssistenteBanco assistente = AiServices.builder(AssistenteBanco.class)
    .chatModel(model)
    .tools(new FerramentasBanco(contaService))
    .chatMemoryProvider(id -> MessageWindowChatMemory.withMaxMessages(20))
    .maxToolCallingRoundTrips(8)                              // teto de loop (padrão 100)
    .toolExecutionErrorHandler((error, ctx) -> {             // NUNCA vaze e.getMessage() ao modelo
        log.error("Tool falhou", error);                     // causa real (o Throwable do handler) só no log do servidor
        return ToolErrorHandlerResult.text("A operação não pôde ser concluída agora.");
    })
    .hallucinatedToolNameStrategy(req ->
        ToolExecutionResultMessage.from(req, "Error: no tool called " + req.name()))
    .build();
★ Ponto de atenção verificado (armadilha do null) — Na linha 1.x, um parâmetro-objeto obrigatório que falta é passado como null SILENCIOSAMENTE (só primitivos faltando lançam ToolArgumentsException). Num banco, isso significa que um accountId nulo pode entrar na sua tool sem erro. Valide objetos críticos defensivamente dentro do corpo da tool — não confie no @P(required=true) para garantir presença. (Isso, e os padrões de erro acima, estão marcados para mudar no LangChain4j 2.0 — reconfirme na sua versão fixada.)

Parte B · 5 — ReturnBehavior, registro e tools dinâmicas

  • ReturnBehavior (@Experimental): TO_LLM (padrão — resultado volta ao modelo, loop continua) vs. IMMEDIATE/IMMEDIATE_IF_LAST (curto-circuita e devolve o resultado direto ao chamador; exige tipo de retorno Result no método do AI Service). Qualquer erro de tool desliga o retorno imediato.
  • Registro — .tools(Object...) / .tools(Collection) é o caminho recomendado (anotações). Existem também .tools(List<AiServiceTool>) e .tools(Map<ToolSpecification, ToolExecutor>) para o caminho programático.
  • Tools dinâmicas — .toolProvider(ToolProvider) resolve a seção 7 ("muitas tools"): você expõe só as tools relevantes ao turno (o tool retrieval/RAG-sobre-tools), em vez de despejar dezenas de descrições na mesa. Conecta com MCP (Model Context Protocol / Protocolo de Contexto de Modelo). A mecânica completa — e o caminho embutido .toolSearchStrategy(...) — está logo abaixo, em §B·6.

Parte B · 6 — Tool retrieval na prática: expor só as tools certas por turno

A seção A·7 prometeu "recupere, por similaridade, só as tools relevantes". Aqui está como — e há dois caminhos reais no 1.16.2, que embutem textos diferentes. Saber qual é qual é a lição.

Caminho A — o mecanismo embutido (ToolSearchStrategy)

O framework já traz RAG-sobre-tools pronto, mas guiado pelo modelo: cada tool nasce com SearchBehavior.SEARCHABLE e — quando você anexa uma estratégia de busca — fica oculta até ser encontrada (sem estratégia anexada, tudo continua visível como nas lições anteriores). A estratégia injeta um meta-tool, tool_search_tool; o modelo, ao perceber que precisa de uma capacidade, formula uma busca e a chama. É a query do modelo que é embutida — não a mensagem do usuário.

// (A) embutido: o framework injeta o meta-tool "tool_search_tool"
EmbeddingModel embeddings = BedrockCohereEmbeddingModel.builder()
    .model("cohere.embed-multilingual-v3")        // multilíngue → cobre pt-BR
    .build();

AssistenteBanco assistente = AiServices.builder(AssistenteBanco.class)
    .chatModel(model)                             // BedrockChatModel
    .tools(todasAsToolsDoBanco)                   // dezenas; SEARCHABLE por padrão
    .toolSearchStrategy(new VectorToolSearchStrategy(embeddings))   // @Experimental
    .build();

Defaults verificados (1.16.2): maxResults=5, minScore=0.0, cacheEmbeddings=true (as descrições embutidas são cacheadas; a query, não). Mantenha tools sempre necessárias (ex.: getSaldo) como ALWAYS_VISIBLE.

Caminho B — o ToolProvider escrito à mão (o seu modelo mental)

O caminho "embute a mensagem do usuário e devolve o top-k" — que é o que a maioria imagina — você escreve. Não há exemplo oficial dele; você o monta a partir de primitivos verificados:

// (B) hand-rolled: VOCÊ recupera por similaridade com a mensagem do usuário
ToolProvider provider = new ToolProvider() {
    @Override public ToolProviderResult provideTools(ToolProviderRequest request) {
        String pergunta = request.userMessage().singleText();         // a mensagem crua
        var achadas = store.search(EmbeddingSearchRequest.builder()
                .queryEmbedding(embeddings.embed(pergunta).content())
                .maxResults(5).build());
        ToolProviderResult.Builder r = ToolProviderResult.builder();
        for (var m : achadas.matches()) {
            r.add(specDaTool(m), executorDaTool(m));   // ToolSpecification + ToolExecutor
        }
        return r.build();
    }
    @Override public boolean isDynamic() { return true; }   // re-avalia a cada chamada do loop
};
AiServices.builder(AssistenteBanco.class).chatModel(model).toolProvider(provider).build();

★ Insight — quem dispara a busca muda tudo. Em (A) o modelo decide buscar e escreve a query (busca tardia, no meio do raciocínio); em (B) você recupera a partir da mensagem do usuário antes de o modelo agir (busca antecipada). Por isso o isDynamic() importa em (B): retornando false (o padrão), o provideTools roda uma vez no início da invocação; retornando true, re-seleciona a cada chamada do loop, conforme a conversa evolui.

⚠️ Honestidade de plataforma. ToolProvider, ToolSearchStrategy e VectorToolSearchStrategy pendem só de interfaces (ChatModel, EmbeddingModel) — então BedrockChatModel + BedrockCohereEmbeddingModel encaixam por construção. Mas o tool-search de ponta a ponta não foi verificado exercitado sobre o Bedrock nesta pesquisa, e a API de busca é @Experimental (pode mudar). Trate como "encaixa por construção", não como "visto rodando".

Há ainda uma terceira forma de filtrar o que o modelo vê — por ativação, não por busca vetorial: as skills, construídas sobre o mesmo ToolProvider. É a Lição 20.

Parte B · 7 — O que os fornecedores confirmam (valida a Parte A)

A pesquisa cruzou a doc oficial com guias da Anthropic e da OpenAI — e eles convergem com os princípios da Parte A:

  • Anthropic ("Writing tools for agents"): "More tools don't always lead to better outcomes"; consolide tools em vez de envolver cada endpoint (ex.: transfer_funds em vez de list_accounts+validate+create_transfer); retorne só informação de alto sinal; descreva a tool "como você descreveria para um novo contratado". A precisão de seleção degrada bastante acima de ~30–50 tools → use tool search/RAG-sobre-tools.
  • OpenAI (function calling): "Aim for fewer than 20 functions available at the start of a turn" (sugestão leve); strict: true (Structured Outputs) garante conformidade de esquema; "don't make the model fill arguments you already know" (= nosso @ToolMemoryId!); "use enums... to make invalid states unrepresentable".

★ Insight — Ou seja: os ~80% de tuning em descrições (seção 3), a consolidação (seção 1) e a identidade-pelo-servidor (seções 4 e B.3) não são opinião — são consenso dos dois maiores fornecedores e da doc do LangChain4j.

Parte B · 8 — Pendência honesta para o Micronaut

O @P tem uma sutileza: sem a opção -parameters do javac, a reflexão vê arg0/arg1 em vez do nome real do parâmetro — por isso Quarkus e Spring ativam -parameters por padrão. Para o Micronaut (DI de compilação), é preciso confirmar se o annotation processor já preserva os nomes ou se você deve usar @P(name = "..."). Não vou afirmar sem checar — fica para o módulo do Micronaut, com código verificado.

Pratique

Quiz — recupere de memória, sem reler

1. Na tag 1.16.2, o que é @Tool.value() e qual o tipo dele?

@Tool.value() é a DESCRIÇÃO e é um String[] (multilinha). O nome é @Tool.name(), que por padrão é o nome do método.

2. Qual é o risco bancário do comportamento-padrão quando uma @Tool lança exceção na linha 1.x?

AiServices captura e envia e.getMessage() ao modelo. Sobrescreva com toolExecutionErrorHandler e devolva mensagem sanitizada; logue a causa real via o Throwable error do handler.

3. No tool retrieval embutido (VectorToolSearchStrategy), que texto vai para a busca vetorial?

No mecanismo embutido (caminho A), quem dispara a busca é o modelo, via o meta-tool tool_search_tool; o texto embutido é a query que ele formula. Embutir a mensagem crua do usuário é o padrão hand-rolled (caminho B), num ToolProvider que você escreve.
