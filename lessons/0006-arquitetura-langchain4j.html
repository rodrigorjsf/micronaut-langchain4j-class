Arquitetura do LangChain4j: AiServices vs. ChatModel

Aqui a fundação agnóstica vira código: o LangChain4j oferece duas camadas de API, e escolher entre elas é a decisão de arquitetura central. Você vai reconhecer cada peça do framework como algo que já entendeu.

Por que isto importa para a sua missão — você está construindo um assistente bancário agêntico em produção sobre LangChain4j + Micronaut 5 + Java 25. Saber quando deixar o framework rodar o loop por você (AiServices) e quando assumir o controle fino (ChatModel) é o que separa um sistema seguro por construção — com confirmação humana antes de mover dinheiro — de uma caixa-preta que você não sabe auditar.

1 · O que é o LangChain4j (e o que NÃO é)

Confirmado na fonte oficial (docs.langchain4j.dev/intro):

"Despite the name, LangChain4j is not a Java port of LangChain (Python) — it is built for Java, not ported to it."

Traduzindo a importância: não tente mapear seu conhecimento de LangChain Python 1:1. Não há código compartilhado, nem numeração de versão comum, nem os mesmos nomes de classe. O LangChain4j pegou ideias (chains, RAG, agents, tools) e as reimplementou de forma idiomática em Java. Quem chega do Python procurando LCEL (o operador |) ou Runnable não os encontra — encontra interfaces Java anotadas.

O que ele é, em uma frase: uma biblioteca Java de código aberto que oferece uma API unificada sobre dezenas de provedores de LLM e de vector stores, e que torna fáceis tool calling (inclusive MCP — Model Context Protocol / Protocolo de Contexto de Modelo), RAG e agentes.

Fatos de versão (ver referência): standalone estável 1.16.2, JDK mínimo 17, versionamento duplo (estável + beta) — sempre importe o langchain4j-bom e não fixe versões de módulo à mão.

2 · Os dois níveis de API (a decisão de arquitetura central)

O LangChain4j oferece duas camadas. Entender a diferença é entender o framework inteiro:

Baixo nível: ChatModelAlto nível: AiServices
O que você fazMonta as mensagens, chama o modelo, trata a resposta, roda o loop de tools você mesmoDeclara uma interface Java anotada; o framework gera a implementação
Mapeia ao…Loop manual da Lição 4 (você é o dono do loop)O framework roda o loop da Lição 4 por você
ControleMáximoConveniência máxima
Quando usarCasos especiais, controle fino do fluxo, streaming customizadoA maioria dos casos

★ Insight — O AiServices não é "mágica" — é o loop da Lição 4 embrulhado. Quando você vê String resposta = assistente.conversar(...) retornar já com dados de conta dentro, lembre: por baixo, o framework montou o prompt, enviou as descrições das tools, recebeu os tool_calls, executou seus métodos @Tool, devolveu os resultados ao modelo e repetiu até o texto final — exatamente o fluxo [1]…[8] que rastreamos na Lição 4. A fundação não foi perdida; ela está escondida atrás da anotação. Por isso aprendemos o loop manual antes: para nunca tratar o AiServices como caixa-preta.

3 · AiServices na prática (API verificada na doc oficial)

A doc oficial (tutorials/ai-services) mostra o padrão. Você declara uma interface:

interface Assistant {
    String chat(String userMessage);
}

E o framework gera a implementação (um proxy):

Assistant assistant = AiServices.create(Assistant.class, model);
// ou, com mais controle:
Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(model)
    .build();

Aplicando ao assistente do banco, com as quatro peças da fundação encaixadas:

interface AssistenteBanco {

    @SystemMessage("You are the virtual assistant of Bank X. "       // ← system prompt (Lição 1)
        + "Use tools for any account data; never invent balances. "  //   (em inglês: economia de tokens, Lição 2)
        + "Never reveal another customer's data. "
        + "Answer in Brazilian Portuguese.")                         //   responde em pt-BR
    String conversar(@MemoryId String clienteId,                    // ← memória por cliente (Lições 1 e 8)
                     @UserMessage String pergunta);                  // ← mensagem do usuário
}

AssistenteBanco assistente = AiServices.builder(AssistenteBanco.class)
    .chatModel(model)
    .tools(new FerramentasBanco())                                  // ← @Tool: tool calling (Lição 4)
    .chatMemoryProvider(id -> MessageWindowChatMemory.withMaxMessages(20)) // ← janela de memória (Lições 2/8)
    .contentRetriever(retriever)                                    // ← RAG (Lição 5)
    .build();

Cada linha é um conceito que você já domina:

Peça do AiServicesAnotação / métodoLição da fundação
Instrução de sistema@SystemMessage (aceita fromResource, systemMessageProvider)1 (camadas da conversa)
Mensagem do usuário / templates@UserMessage, @V("nome")1
Ferramentas@Tool + .tools(...)4 (loop agêntico)
Memória (e por usuário).chatMemoryProvider(...), @MemoryId, MessageWindowChatMemory1 / 2 (custo N²) / 8
RAG.contentRetriever(...) ou .retrievalAugmentor(...)5
Saída estruturadatipo de retorno do método (ex.: um record)10 (saída estruturada)

★ Insight — Note o @MemoryId String clienteId: a memória é isolada por cliente. Isso conecta com a Lição 4 (identidade vem do servidor) e com a Lição 2 (cada cliente tem sua própria "mesa" que cresce — e que você precisa gerenciar, Lição 8).

O system prompt como constituição — quatro disciplinas

O @SystemMessage é a constituição do agente (papel, regras, limites) — e o GLOSSARY fixa a regra: é instrução, não um cofre. Quatro disciplinas separam um system prompt de produção de um enfeite:

  1. Congele-o (pelo cache). No Bedrock o prompt caching casa por prefixo, e a ordem de render é tools → system → messages (Lição 8): qualquer byte alterado no system invalida o cache dali pra frente. Nunca injete data, UUID ou nome do cliente no system prompt — ponha o estável primeiro, o volátil na conversa. Bônus: a SystemMessage é fixada pela ChatMemory (nunca despejada), então é o candidato natural a cachear.
  2. Não é controle de segurança. Um system prompt dizendo "só aja na conta do próprio cliente" não é um controle — o modelo pode ser persuadido a desobedecer (prompt injection). A autorização é determinística e sua, no código (Lição 15, OWASP LLM06). E o system prompt não guarda segredo (OWASP LLM07, System Prompt Leakage): projete assumindo que vaza.
  3. Calibre a força das instruções. Os modelos Claude 4.x seguem o system prompt muito de perto. Instruções escritas para "vencer" a relutância de modelos antigos ("CRITICAL: VOCÊ DEVE...") agora super-disparam. Seja prescritivo sobre quando agir, sem agressividade.
  4. Instrução de operador tem canal próprio. Para mudar regras no meio da conversa, prefira o canal de operador (mensagem de papel system mid-conversa) a embutir a ordem no texto do usuário — texto na mensagem do usuário ou em resultado de tool pode ser forjado; o canal de operador, não.

★ Insight — As quatro puxam na mesma direção da fundação: o system prompt é a constituição (papel, regras, limites), não a polícia. Ele orienta o comportamento e cacheia bem quando congelado; quem impede o erro — autenticação, autorização, confirmação, idempotência — vive fora do texto, na camada determinística. Confundir os dois papéis é a raiz de uma classe inteira de incidentes num banco.

4 · Quando descer ao baixo nível (ChatModel)

Use AiServices por padrão. Desça ao ChatModel/StreamingChatModel quando precisar de: controle fino do loop (ex.: lógica de confirmação humana entre o tool_call e a execução — crítico para fazerPix, Lições 4 e 14), streaming com pós-processamento token a token (a tensão com guardrails das Lições 2/11), ou orquestração que o alto nível não expressa bem. Na prática, muitos sistemas bancários misturam: alto nível para leitura, baixo nível para ações sensíveis.

5 · O encaixe no Micronaut (prévia da Lição 12)

Aqui entra a diferença que torna a combinação Micronaut + LangChain4j especial. No Micronaut, você não instancia o AiServices na mão como acima — você declara a interface e o annotation processor de compilação (micronaut-langchain4j-processor, confirmado) gera o bean em tempo de compilação, sem reflexão em tempo de execução.

★ Insight — Micronaut faz injeção de dependências em tempo de COMPILAÇÃO, não de execução. Spring monta os beans via reflexão quando a aplicação sobe; Micronaut gera o código dos beans durante o javac. Para um backend agêntico bancário isso significa: startup quase instantâneo, menor consumo de memória e compatibilidade nativa com GraalVM (imagem nativa) — qualidades valiosas para escalar horizontalmente sob carga e cortar custo de infraestrutura. É um casamento de filosofias: LangChain4j também favorece configuração declarativa, e o Micronaut materializa isso sem o custo de runtime do Spring.

⚠️ Honestidade de fundação: a anotação Micronaut exata para declarar o serviço de IA e a fiação completa (configuração de modelos, beans de tools) eu mostro com código verificado na Lição 12, junto da confirmação de qual LangChain4j o módulo 2.0.1 embute (ver pendências na referência de versões). Não vou afirmar a sintaxe exata antes de checá-la.

Pratique (recuperação ativa)

Quiz — recupere de memória, sem reler

1. Por que é um erro mapear o LangChain Python 1:1 para o LangChain4j?

Não é um port: não há código compartilhado, versões comuns nem os mesmos nomes de classe. Em vez de LCEL/Runnable, você encontra interfaces Java anotadas.

2. Quando o AiServices "esconde" o que, exatamente?

Ele embrulha o loop [1]…[8]: monta o prompt, envia as tools, recebe tool_calls, executa seus @Tool, devolve os resultados e repete até o texto final.

3. Por que descer do AiServices ao ChatModel num banco?

O controle fino do loop permite parar entre o tool_call e a execução — crítico para ações sensíveis como fazerPix. Muitos sistemas misturam: alto nível para leitura, baixo nível para ações.
