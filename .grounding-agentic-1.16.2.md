## Human-in-the-loop — DEFINITIVELY EXISTS (refutes earlier verification-debt doubt)
Two forms at tag 1.16.2:
1. Programmatic: dev.langchain4j.agentic.workflow.HumanInTheLoop (record + HumanInTheLoopBuilder).
   Builder: responseProvider(Function<AgenticScope,?> | Supplier<?>), outputKey, description, async(boolean),
   inputKey(Class,String), inputKeys(...), listener(AgentListener).
   The @Agent description: "An agent that asks the user for missing information".
2. Declarative: @dev.langchain4j.agentic.declarative.HumanInTheLoop on a static method.
- Docs (tutorials/agents.md line 2263): HITL lets system "ask user's input for missing information or APPROVAL BEFORE PROCEEDING with certain actions." => this is the built-in confirmation hook for money-moving ops.
- It is modeled as a "special non-AI agent" (deterministic). responseProvider is your app code (console, queue, REST callback). async(true) recommended.
- Recoverability: with an AgenticScopeStore configured, workflows including HITL steps can pause/resume across crash/restart (line 2464).

## Money-moving confirmation & idempotency
- No money/confirm/idempotency-specific class in agentic module (grep of tree @1.16.2 = none).
- Confirmation = use HumanInTheLoop (approval) OR a deterministic tool returning "needs confirmation" + a gate; framework-supported via HITL agent.
- Idempotency = ARCHITECTURE pattern, NOT a langchain4j API. Implement idempotency keys at the @Tool / backend layer (BankTool). Tutorial's BankTool/@Tool withdraw/credit are plain methods — no idempotency built in.
- Tutorial uses a BANKING supervisor example: bankSupervisor.invoke("Transfer 100 EUR from Mario's account to Georgios' one"), subagents WithdrawAgent/CreditAgent/ExchangeAgent over BankTool/ExchangeTool with @Tool/@P.
